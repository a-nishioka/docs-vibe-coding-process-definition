# CODING_STANDARDS.md — コーディング規約

> Claudeは実装時にこの規約を遵守すること。
> 規約に迷った場合は実装を止めてユーザーに確認すること。

---

## 命名規則

| 対象 | 規則 | 例 |
|---|---|---|
| 変数・関数 | camelCase | `getUserName()` |
| クラス・型 | PascalCase | `UserProfile` |
| 定数 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| ファイル名 | kebab-case | `user-profile.ts` |
| CSS クラス | kebab-case | `user-card` |

> ※ プロジェクトの言語・フレームワークに合わせて修正すること

---

## ファイル構成ルール

- 1ファイル1責務を原則とする
- ファイルは **300行以内** を目安とする（超える場合は分割を検討）
- 関数は **50行以内** を目安とする
- インポートの順序:
  1. 標準ライブラリ
  2. 外部ライブラリ
  3. 内部モジュール（絶対パス）
  4. 内部モジュール（相対パス）

## コメントルール

### 書くべきコメント
- **なぜ**そうしているか（Why）
- 複雑なビジネスロジックの説明
- 外部仕様・APIの制約に関する注意点
- TODO・FIXME（内容と担当を明記）

```typescript
// ✅ 良いコメント: 理由を説明している
// APIが500件以上のリクエストを拒否するため、バッチサイズを制限する
const BATCH_SIZE = 499;

// ❌ 悪いコメント: コードを読めばわかる内容
// ユーザー名を取得する
const userName = user.name;
```

### 書かないコメント
- コードを読めば明らかな内容
- 古くなった（実態と乖離した）コメント

## エラーハンドリング

- エラーは**握り潰さない**（`catch` 内で何もしない実装は禁止）
- エラーメッセージはユーザーにとって意味のある内容にする
- 予期しないエラーはログを残す

```typescript
// ✅ 良い例
try {
  await fetchUser(id);
} catch (error) {
  console.error('ユーザー取得に失敗しました:', error);
  throw new Error(`ユーザー(id: ${id})の取得に失敗しました`);
}

// ❌ 悪い例
try {
  await fetchUser(id);
} catch {
  // 何もしない
}
```

## 型定義（TypeScript使用時）

- `any` の使用を原則禁止とする
- `unknown` を優先的に使用し、型ガードを実装する
- 型定義は `types/` ディレクトリに集約する

## テストコード

- テストファイルは実装ファイルと同階層に配置する（`*.test.ts`）
- テスト名は「〜すること」「〜のとき〜を返すこと」の形式で書く
- 1テスト1アサーションを原則とする

```typescript
// ✅ 良いテスト名
it('メールアドレスが不正な場合にエラーを返すこと', () => { ... });

// ❌ 悪いテスト名
it('test1', () => { ... });
```

## Git コミットルール

コミットメッセージは以下のプレフィックスを使用する：

| プレフィックス | 用途 |
|---|---|
| `feat:` | 新機能 |
| `fix:` | バグ修正 |
| `docs:` | ドキュメント変更 |
| `refactor:` | リファクタリング |
| `test:` | テストの追加・修正 |
| `chore:` | ビルド・設定変更 |

```
# 例
feat: ユーザープロフィール編集機能を追加
fix: ログアウト時にセッションが残る問題を修正
```
